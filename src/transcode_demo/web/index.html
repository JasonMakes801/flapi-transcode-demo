flapid process lifecycle

The demo keeps one persistent search flapid (probe + scan) and launches an ephemeral render flapid per worker. They tear themselves down on the idle timeout — or kill them here. Scan/Transcode will spin up new ones automatically as needed.

What this is

A reference integration showing you can drive the FilmLight render API (FLAPI) from your own application — mass-transcoding camera originals to editorial proxies, with a small web GUI on top. The API is yours to build on.

The stack — deliberately tiny

Backend
Python + FastAPI (one process)
Frontend
Alpine.js, served by FastAPI — no Node, no npm, no build step
Render
FilmLight FLAPI; each worker launches its own child flapid per clip, torn down after ( , set TCDEMO_WORKERS)
Throughput
scales with GPUs, not CPU cores: debayer + colour + encode run on the GPU, so one render worker per GPU. On a multi-GPU render node the workers render in parallel; on a single-GPU machine (e.g. any Apple Silicon Mac) they funnel to one GPU and run one at a time — hence the default of 1.
Colour
GPU debayer + transform from camera space → a selectable display colourspace (Rec.1886, sRGB, P3, DCI-P3…)
Run
./bootstrap.sh once, then ./run.sh → http://localhost:8080
